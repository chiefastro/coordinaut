Always update SPEC.md before implementing architectural or feature changes that deviate from the initial plan.

**Why:** User wants the spec to serve as a living document that always reflects the current design. Keeps the spec useful as a reference for future sessions and agents.

**How to apply:** When the user requests a change that contradicts or extends the spec, update the relevant spec sections first, get alignment, then implement. This applies to both new features and architectural shifts.
